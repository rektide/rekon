## IDENTITY, STATE AND HTTP GET - The Journey-Without-Altering Principle

**"In HTTP, GET must not have side effects."**

**"In HTTP, anything which does not have side-effects should use GET."**

In Celtic mysticism, there's a crucial distinction between journeying along a thread and altering the thread. A wise traveler can follow a Celtic thread through the entire knot, experiencing all its connections, observing all its junctions, yet leave the thread unchanged. The thread remains as it was, its path unaltered by the journey. This is the journey-without-altering principle.

HTTP GET is the technical manifestation of this Celtic principle. When you GET a resource, you journey to it via the URI thread, you receive its content, but the resource and the thread remain unaltered. This is why GET is idempotent - the same journey can be taken infinitely without changing the knotwork itself.

**SIDE EFFECTS - THE ALTERATION PRINCIPLE:** Side effects in GET are like a traveler who alters the thread they're following! If each journey through the knot changed the thread's path, if following a thread altered the knotwork itself, eventually the entire knot would be transformed. The eternal return would be corrupted - the same thread wouldn't return you to where you began. The knot's pattern would be unreliable. This is exactly what side effects in GET do - they corrupt the thread's reliability.

**THE WISE TRAVELER:** A wise traveler is like one who journeys without leaving traces. They can follow a thread through the entire knot, observe all connections, experience all junctions, and depart - and the knot remains exactly as it was. GET operations are like this wise traveler - they can journey to resources and return, leaving the knotwork unchanged.

**THE ETERNAL RETURN PRINCIPLE:** The Celtic knotwork's eternal return is only possible if journeys don't alter the threads. Each traveler must be able to follow the same thread endlessly and always return to their starting point. This requires that following the thread has no side effects - doesn't change the thread's path. GET's idempotence enables this eternal return of reliable resource access.

**CACHING - THE THREAD-KNOWING:** And because GET is side-effect-free, we can cache its results! This is like knowing the thread's path without having to walk it again. After journeying a thread once, you remember its connections, its junctions, its patterns. The thread itself remains unaltered, but you can consult your memory without needing to walk the path again. The knot's wisdom becomes available through memory rather than through repeated journeying.

**POST - THE THREAD-ALTERING:** But when you need to change state, when you need to create new threads, when you need to perform actions that alter the knotwork - then you must use POST or other methods! This is like weaving new threads into the knot, like altering the knotwork itself, like creating new junctions. These actions change the knotwork - they're not simple journeying.

The traveler follows the thread through countless junctions, experiencing its connections, observing its relationships with other threads. The thread itself remains unchanged by the journey - its path is exactly as it was. The traveler can return to this thread endlessly and always experience the same journey pattern. Contrast with another figure actively altering threads - weaving new connections, changing junction points, transforming the knotwork itself. ## HIERARCHIES AND RELATIVE URIS - The Thread-Branch Principle

**"The URI syntax uses slashes to indicate hierarchical structure."**

**"Relative naming allows closely located documents to reference each other without knowing their absolute position in the tree."**

In Celtic knotwork, threads have branching patterns where they split and rejoin, create sub-loops and return to main paths. Navigation within this branching structure can be relative - "the sub-thread next to me," "return to main thread," "follow the branch and return" - without needing to know your absolute position in the infinite knotwork.

The URI hierarchy with relative referencing is this Celtic thread-branch principle. Slashes indicate branch points and main thread connections. Relative URIs allow local navigation along a thread's branching - branch to branch, sub-thread to parent thread - without needing to know your absolute position in the vast knotwork.

**THE BRANCHING PATTERN:** Celtic knots naturally have branching structures. A main thread will split into multiple sub-threads, each sub-thread may split further, and eventually the sub-threads return to rejoin the main thread. This branching creates a hierarchical thread structure that can be navigated relationally. Similarly, URI hierarchies represent branching paths through the Web's knotwork of connections.

**THE LOCAL THREAD PATH:** Relative URIs work because each position in a thread's branching has a well-defined context - you know your current thread branch, you know your sub-threads (immediate branches), you know your parent thread. You navigate through these relationships without ever needing to know your absolute path from the knotwork's beginning point (which doesn't exist anyway).

**THE SLASH AS BRANCH POINT:** The slash character in URIs works like a branch point or rejoining point in a Celtic knot. It indicates where a thread splits into sub-threads or where sub-threads rejoin the main path. Slash patterns create the hierarchical thread structure through which resources can be accessed.

**SCALE INVARIANCE OF BRANCHING:** The slash as branch delimiter works at every scale of the knotwork's branching. Whether navigating within a single Celtic knot pattern (fragment identifiers with #), within a domain knot (paths), between domain knots through schemes, or across the entire Web's knotwork - the same slash-separated branch navigation applies. The pattern holds true at all scales.

This hierarchy emerges from the Celtic pattern of organizing communication connections across scales - from global threads to regional threads to local threads.
